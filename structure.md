# Документация структуры проекта CRM

Проект **STO CRM** построен как модульное Python-приложение с интегрированным одностраничным веб-интерфейсом (SPA) на стороне клиента, хранением данных в реляционной СУБД SQLite и возможностью сборки в единый автономный исполняемый файл `.exe` для операционной системы Windows.

Ниже приведено детальное описание файловой структуры и функционального назначения каждого компонента проекта.

---

## 2. Назначение основных модулей пакета `sto_crm`

* **`main.py`**
  Инициализирует глобальный экземпляр `Runtime`, конфигурирует и запускает HTTP-сервер. При экспорте через PyInstaller служит отправным скриптом для упаковки.

* **`sto_crm/__init__.py`**
  Предоставляет фасад совместимости (`_StoCrmFacade`), чтобы старый импорт `import sto_crm` продолжал работать без изменений в тестах и легаси-скриптах. Автоматически публикует атрибуты внутренних модулей в глобальное пространство пакета.

* **`sto_crm/database.py`**
  Отвечает за подключение к SQLite с оптимальными PRAGMA (WAL режим, busy_timeout=30 секунд, mmap, нормальная синхронизация) для высокой производительности на ARM и x86. Определяет схему таблиц: клиенты, автомобили, запчасти на складе, заказы, работы/детали, записи в календарь. Предоставляет менеджер `write_db()`, сериализующий транзакции с помощью `BEGIN IMMEDIATE` для исключения гонок.

* **`sto_crm/validation.py`**
  Обеспечивает строгость бизнес-логики: форматирование телефонов, проверку контрольной суммы VIN, валидацию года выпуска автомобиля, финансовые лимиты и проверку пересечения записей в календаре по времени в пределах +-15 минут для одного мастера или подъемника.

* **`sto_crm/services.py`**
  Точка выполнения транзакционных бизнес-процессов: создание, обновление или мягкое удаление записей. Содержит логику блокировки изменения закрытых/отмененных заказов, списания и возврата остатков на складе, контроля уникальности VIN с учетом soft-delete.

* **`sto_crm/queries.py`**
  Концентрирует логику чтения данных с фильтрацией, пагинацией и гибким текстовым поиском (`LIKE %q%`). Вычисляет финансовые итоги заказ-нарядов с учетом скидки, налоговой ставки и статуса согласования отдельных позиций (отложенные и отказанные строки исключаются из кассы и не резервируют склад).

* **`sto_crm/reports.py`**
  Генерирует аналитические срезы:
  * Индекс здоровья смены (просроченные обязательства, незакрытые визиты, критический дефицит склада).
  * Распределение воронки заказов по статусам.
  * План закупки (список деталей, у которых остаток ниже минимального объема).
  * Выручку, маржу, средний чек, конверсию смет и список VIP-клиентов.
  * Персональный план смены (фокусирует мастера на задачах «сделать сегодня» и «проверить клиентов»).

* **`sto_crm/updates.py`**
  Осуществляет скачивание метаданных и релизов с GitHub. Отвечает за:
  * Валидацию скачиваемого EXE (заголовки, размер, хэш SHA-256).
  * Ротацию резервных копий SQLite БД (хранение до `MAX_BACKUP_FILES` файлов совокупным размером не более `MAX_BACKUP_TOTAL_BYTES`).
  * Резервное копирование текущего бинарника перед обновлением с поддержкой автоматического отката (rollback) в случае сбоя.

* **`sto_crm/web.py`**
  Маршрутизирует HTTP-запросы.
  * Обрабатывает статические страницы и ассеты.
  * Реализует REST API `/api/appointments`, `/api/orders`, `/api/customers`, `/api/vehicles`, `/api/inventory`.
  * Валидирует CSRF-токены для мутирующих запросов и защищает от CORS атак (проверка разрешенных Origin).

* **`sto_crm/assets/`**
  Содержит веб-интерфейс приложения:
  * `index.html`: Точки встраивания стилей и скриптов.
  * `app.css`: Реализация темы оформления (светлая/темная), переменные размеров, сетка плиток KPI, стили печатной формы с поддержкой CSS-разрывов страниц (`page-break-inside: avoid`).
  * `app.js`: Клиентский SPA-движок на чистом JS. Включает навигацию по хэшам страниц, управление модальными формами, командную палитру Ctrl+K для быстрого переключения разделов и поиска команд, ведение черновика сметы заказ-наряда, оффлайн-режим с кэшированием структуры и дебаунсингом запросов.

## 3. Архитектурные особенности проекта

1. **Монолитный дистрибутив**: Все frontend-файлы (`app.js`, `app.css`, `index.html`) при компиляции или запуске динамически считываются и поставляются сервером в виде встроенных ресурсов. Это позволяет распространять СРМ одним исполняемым файлом, не требующим наличия NodeJS или веб-сервера (Nginx/Apache) в операционной системе Windows.
2. **Слой совместимости с Legacy-кодом**: Использование метакласса `_StoCrmFacade` в `sto_crm/__init__.py` симулирует единое пространство имен оригинального монолитного модуля `sto_crm.py` для сохранения работоспособности старых интеграционных тестов.
3. **Безопасность песочницы**: Все пути работы с файлами базы данных, логов и резервных копий нормализуются с проверкой на выход за пределы папки пользователя `%LOCALAPPDATA%` или домашней папки (защита от атаки Path Traversal).
4. **Отказоустойчивость бэкапов**: При записи новых данных сначала генерируется транзакция на запись, а при обновлении приложения сначала производится копирование рабочей базы, прогон контрольных сумм скачанного файла и только в случае успеха — подмена основного бинарника.
