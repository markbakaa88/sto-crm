# Отчет по техническому и UI/UX аудиту проекта СТО CRM

## 1. Введение и общая оценка
СТО CRM представляет собой отзывчивое локальное веб-приложение для автосервисов, реализованное на стыке Python (бекенд с базой данных SQLite) и ванильного JavaScript (фронтенд в виде SPA-бандла).

В ходе аудита была проанализирована структура проекта, конфигурация линтеров, качество типов MyPy, тестовое покрытие (186 тестов, >91% покрытия), структура ассетов фронтенда (`app.js`, `app.css`, `index.html`) и UX/UI паттерны.

---

## 2. Глубокий технический аудит (Python, API, DB, Tests)

### Выявленные сильные стороны:
1. **Строгая типизация**: Проект использует аннотации типов Python на высоком уровне. Инструмент `mypy` проходит с 0 ошибок.
2. **Высокое покрытие тестами**: Интеграционное и юнит-тестирование покрывает 91% кодовой базы, включая маргинальные случаи обновления приложения, обработку ошибок сети и валидацию данных.
3. **Изоляция транзакций**: Запись в базу SQLite защищена менеджером контекста `write_db()` с ранним повышением блокировки транзакции (`BEGIN IMMEDIATE`), что предотвращает конфликты блокировки базы данных (`SQLITE_BUSY`) при параллельных запросах.
4. **Безопасность**: Защита от CSRF реализована для всех мутирующих API (`POST`/`PUT`/`DELETE`) с передачей токенов безопасности. Также присутствует строгая фильтрация адресов при загрузке внешних ресурсов.

### Области для технического улучшения:
1. **Обработка устаревших CSP директив**: В файле `index.html` используется CSP, содержащая конструкции `'strict-dynamic'`, небезопасна при использовании в старых браузерах без явного указания фоллбеков (`'unsafe-inline'`).
2. **Жестко зашитые лимиты резервных копий**: Удаление старых бэкапов выполняется в `sto_crm/updates.py` на основе фиксированных констант `MAX_BACKUP_FILES = 20` и `MAX_BACKUP_TOTAL_BYTES`, что затрудняет администрирование крупным СТО.
3. **Управление фоновыми процессами обновлений**: Во время установки обновлений из GitHub Releases (`install_update_from_github`) прибивание / перезагрузка процесса локального сервера выполняется через `sys.exit` или `server.shutdown`, работающие в отдельном потоке. Это может приводить к зомби-процессам под Windows, если десктопный менеджер окон (например, WebView2 / Electron узел) не успеет корректно закрыть зависимости.

## 3. Аудит UI/UX и доступности (CSS, JS, Glassmorphism, a11y)

### Стек Glassmorphism и Визуальный Дизайн:
1. **Злоупотребление селекторами `!important`**: В `app.css` в секции `Premium UI & Glassmorphism Enhancements` правила для классов `.section-card`, `.panel`, `.modal` и др. переопределяют цвета фона, размытие и тени через `!important`. Это нарушает каскад стилей CSS и затрудняет кастомизацию отдельных окон и элементов (например, опасных предупреждений или темных панелей).
2. **Размытие и производительность (Blur)**: Использование эффектов `backdrop-filter: blur(20px)` на крупных контейнерах (например, на боковой панели `.sidebar`, модальных окнах `.modal` и `.command-palette` одновременно) вызывает лаги отрисовки интерфейса на старых офисных компьютерах, часто используемых в гаражных автосервисах. Оптимально снизить размытие до `blur(8px) - blur(12px)`.
3. **Контрастность (Contrast)**:
   - В светлой теме цвет текста `--ink-muted: #475569` (серый) на слегка прозрачном стеклянном фоне `.section-card` (белый с прозрачностью `0.65` на светло-сером фоне приложения `--bg: #f5f7fb`) не проходит стандарт доступности WCAG AA (минимальный коэффициент контрастности 4.5:1).
   - В темной теме некоторые серые подписи приглушенных элементов (`--disabled-ink: #64748b`) сливаются с темным фоном поверхности.

### Защита от случайных действий и Double Submit:
1. **Блокировка повторных запросов (Double Submit)**:
   - В функциях `saveEntity` и `saveOrder` корректно используется функция `setSaveButtonsBusy(true)`, которая блокирует кнопки сохранения. Однако отсутствует блокировка (или визуальный индикатор) при быстрых кликах по кнопкам удаления (`deleteEntity`), дублирования заказов или при нажатии сочетаний клавиш.
   - Мелкие интерактивные кнопки добавления позиций (`#addService`, `#addPart`) также не дебаунсятся, что позволяет добавлять дубликаты строк при множественном клике.
2. **Случайное закрытие форм**:
   - `closeModal` использует стандартное браузерное окно подтверждения `confirm("Закрыть окно без сохранения изменений?")`. Это выбивается из премиального стиля приложения с эффектом стекла и ломает бесшовный пользовательский опыт. Целесообразно заменить его на кастомное модальное диалоговое окно в стиле glassmorphism.
   - Механизм `shouldKeepModalForEscape` корректно предотвращает закрытие формы при нажатии `Esc` во время ввода в текстовых полях (`textarea`, `select`), но не учитывает динамически добавляемые кастомные дропдауны или интерактивные элементы управления.

### Доступность (a11y) и Управление фокусом:
1. **Фокус при закрытии модальных окон**: При закрытии модального окна на десктопе фокус возвращается на `lastFocusedElement`. Однако отсутствует визуальная рамка фокуса для многих иконок и элементов управления навигации (например, `.brand-mark` при переходе на панель), что мешает клавиатурной навигации (`Tab`).
2. **Скринридеры (Screen Readers)**:
   - В `index.html` и динамических шаблонах в `app.js` используется разметка `aria-live="polite"` на тостах и контейнере `#appStatus`, но не везде объявлен корректный `tabindex` для неинтерактивных областей контента, когда они становятся активными.
   - Элемент `.search-hint` (`/` на клавиатуре) визуально скрыт через `display: none` (`.search-hint { display: none; }`), однако в некоторых состояниях он должен быть доступен для экранных дикторов.

## 4. Пошаговый список задач по улучшению проекта

Сгруппированный перечень взаимонезависимых задач, готовых к выполнению.

### Шаг 1. Оптимизация стилей Glassmorphism и повышение контрастности
- [x] **Рефакторинг CSS**: Избавиться от директив `!important` в секции `Premium UI & Glassmorphism Enhancements`. Использовать более специфичные вложенные селекторы или переопределить базовые дизайн-токены (CSS variables) тёмной и светлой темы.
- [x] **Снижение нагрузки**: Уменьшить радиус размытия `backdrop-filter` с `20px` до приемлемых `10px` для повышения стабильности частоты кадров (FPS) на слабых GPU.
- [x] **Контрастность текстов**:
  - Изменить цвет текста `--ink-muted` на более темный `#2d3748` для светлой темы и `--ink-muted` на `#cbd5e1` для темной темы на поверхностях со стеклянным фоном.
  - Повысить читаемость заблокированных (`disabled`) инпутов, заменив `--disabled-ink` на более контрастный серый.

### Шаг 2. Повышение надежности UI-интерфейса и интерактивности
- [x] **Кастомное подтверждение закрытия**: Написать единую функцию предупреждения о несохраненных изменениях с показом стилизованного под стекло оверлея поверх модального окна вместо стандартного синхронного диалога `confirm()`.
- [x] **Защита от двойных нажатий (Double-click / Debounce)**:
  - Добавить дебаунсинг и визуальную блокировку состояния (`disabled`) для кнопок удаления записи (`deleteEntity`) и быстрого дублирования заказов.
  - Реализовать кратковременное отключение нажатий (например, на `300ms`) на кнопки `#addService`, `#addPart` во избежание случайного дублирования.
- [x] **Предотвращение закрытия при выборе даты**: Исключить закрытие модалок по клавише `Esc`, если открыт нативный календарь выбора даты в браузере (через отслеживание фокуса на элементах `input[type="date"]` и `input[type="datetime-local"]`).

### Шаг 3. Доступность (a11y) и клавиатурное управление
- [x] **Визуальный фокус (Focus Outline)**: Добавить четкую рамку фокуса (`outline-color: var(--brand)`) для интерактивных элементов в боковом меню, кнопках верхней панели (`#commandBtn`, `#refreshBtn`) при управлении клавиатурой (`:focus-visible`).
- [x] **Удержание фокуса (Focus Trap)**: Убедиться, что при открытии модального окна фокус принудительно переводится на первый доступный элемент внутри формы, а при закрытии — возвращается на элемент, вызвавший окно.

### Шаг 4. Технические баг-фиксы и оптимизация кода
- [x] **Обновление CSP**: Добавить резервные директивы для старых браузеров в мета-тег Content-Security-Policy (например, `'unsafe-inline'` вместе с используемыми `nonce` для совместимости).
- [x] **Мягкое завершение фоновых потоков**: Доработать процедуру завершения работы сервера Python (в `sto_crm/cli.py` и `sto_crm/http_server.py`), добавив явный лаг закрытия сетевых соединений (graceful shutdown) при получении команды перехода в оффлайн или перезагрузки для установки обновлений.
