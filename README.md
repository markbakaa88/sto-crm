# СТО CRM

Локальная CRM для автосервиса: один `.exe` запускает сервер на `127.0.0.1`, открывает браузер и хранит данные в SQLite в профиле пользователя.

Исходный код разделен на пакет `sto_crm/` по зонам ответственности (runtime, SQLite, валидация, бизнес-операции, отчеты, HTTP, обновления и frontend-assets), а корневой `sto_crm.py` оставлен тонким совместимым launcher для запуска и PyInstaller.

## Готовый запуск

Скачайте готовый single-file `STO_CRM.exe` из GitHub Releases проекта или запустите локально собранный файл после `./build.ps1`:

```powershell
.\release\STO_CRM.exe
```

Для запуска на чистой Windows ничего устанавливать не нужно: Python, SQLite и runtime-библиотеки упакованы внутрь `.exe`.

При первом запуске `.exe` создаст чистую базу в `%LOCALAPPDATA%\STO_CRM\sto_crm.sqlite3`, поэтому папка с приложением может содержать только один файл. В интерфейсе есть кнопка `Остановить`, которая завершает локальное приложение.

## Обновления через GitHub

Приложение подключено к публичным GitHub Releases репозитория `markbakaa88/sto-crm`. В релиз загружаются готовый билд, checksum и manifest; приложение скачивает только release-assets и не использует GitHub-токены. Механизм обновления работает так:

1. откройте `Обновления` → `Проверить обновления`;
2. CRM получает последний GitHub Release и asset `latest.json`;
3. manifest указывает на готовый `STO_CRM.exe`, его размер и SHA-256;
4. при установке скачанный exe проверяется по размеру, SHA-256 и формату Windows `.exe` (MZ-заголовок);
5. перед заменой создается консистентная резервная копия SQLite-базы в папке `backups` рядом с рабочей БД;
6. текущий `.exe` сохраняется как резервная копия в `%LOCALAPPDATA%\STO_CRM\updates`;
7. приложение закрывается, заменяет exe, при ошибке замены пытается откатиться на резервную копию и запускается снова.

Репозиторий обновлений должен быть публичным: приложение не хранит авторизацию для скачивания обновлений.

Если нужно использовать другой публичный форк/репозиторий без пересборки, задайте:

```powershell
setx STO_CRM_UPDATE_REPOSITORY "owner/repo"
```

При запуске из исходников автоустановка недоступна — используйте `git pull --ff-only`, затем перезапустите `python .\sto_crm.py`.

## Возможности

- клиенты, автомобили, склад и заказ-наряды;
- календарь записи клиентов: дата/время, длительность, статус визита, мастер-приемщик, причина и CSV-экспорт;
- цифровые мульти-точечные осмотры DVI: чек-лист, состояние `норма/внимание/критично`, рекомендации, оценка работ и CSV-экспорт;
- встроенный автономный справочник на 265 производителей и 2588 моделей автомобилей в карточке авто;
- отдельный раздел `Каталог авто`, API `/api/catalog` и CSV-экспорт полного списка производителей и моделей;
- статусы ремонта, приоритеты, мастер-приемщик, механик, сроки;
- работы, складские и вне-складские/заказные запчасти в заказ-наряде с расчетом итогов, скидки, оплаты и долга;
- согласование каждой строки заказ-наряда: согласовано, отложено или отказ; отложенные/отказанные строки не попадают в итог и не списывают склад;
- согласование сметы, фиксация ответственного согласовавшего и даты согласования;
- CRM-задачи для follow-up после визита, персональных сервисных напоминаний, несогласованных смет и возврата отказанных/отложенных работ;
- автоматические CRM-задачи по критичным пунктам осмотра, которые еще не согласованы клиентом;
- предпочтительный канал связи клиента и согласие на сервисные напоминания;
- план следующего обслуживания по дате и пробегу в карточке автомобиля;
- автоматическое списание только складских запчастей при закрытии заказ-наряда и возврат при отмене;
- executive cockpit: индекс здоровья смены, активная воронка заказ-нарядов, просрочки, загрузка на 7 дней, план закупки и загрузка ответственных;
- умный `План смены`: автоматическая приоритизация просроченных заказов, DVI-рисков, несогласованных смет, follow-up, сервисных напоминаний, закупки и записей на сегодня с быстрыми переходами в нужную карточку;
- панель KPI, отчеты, низкие остатки, топ работ, валовая маржа, конверсия смет, стоимость склада и VIP-сегмент клиентов;
- премиальный профессиональный рабочий интерфейс: лаконичная executive-панель, операционный статус-бар смены, nav-бейджи рисков, единые заголовки разделов, темная/светлая тема, переключатель плотности, командная палитра Ctrl+K, адаптивная навигация, быстрые действия, offline-индикатор и улучшенные пустые состояния;
- профессиональная печатная форма заказ-наряда с фирменным hero-блоком, статусами согласования строк, адаптивной версткой и print-стилями;
- CSV-экспорт по токену из интерфейса и консистентная резервная копия базы;
- профессиональный раздел `Обновления`: проверка GitHub Releases, выбор `STO_CRM.exe`, контроль размера/SHA-256, резерв SQLite-базы и текущего exe, безопасный перезапуск;
- защита локальных изменяющих запросов и экспорта от внешних Origin;
- строгая валидация числовых полей и JSON-позиций;
- защита финансовой истории закрытых заказ-нарядов от незаметного изменения;
- автотесты на критичную бизнес-логику, UI-контракты, резервные копии и склад.

## Запуск из исходника

```powershell
python .\sto_crm.py
```

Дополнительные параметры:

```powershell
python .\sto_crm.py --host 127.0.0.1 --port 8780 --db .\my_service.sqlite3 --no-browser
```

Для демонстрационной базы:

```powershell
python .\sto_crm.py --demo
```

## Пересборка `.exe`

Требования для сборки: Windows/PowerShell, установленный Python (или `py` launcher). `build.ps1` устанавливает/проверяет совместимую версию PyInstaller через `pip`, поэтому для первого запуска может понадобиться интернет или заранее прогретый pip-cache.

```powershell
.\build.ps1
```

Скрипт можно запускать из любой директории: все пути вычисляются от папки проекта. Он запускает `py_compile` для launcher, пакета `sto_crm/`, frontend-assets package marker и тестов, полный `unittest`, при необходимости устанавливает совместимую версию `PyInstaller` (`pyinstaller>=6.10,<7`), очищает `build`/`dist`, выполняет чистую PyInstaller-сборку из единого файла спецификации `STO_CRM.spec`, пересоздает `release`, копирует single-file исполняемый файл в `release\STO_CRM.exe` и выполняет smoke-test собранного `.exe` через `/api/health`, `/api/bootstrap` и корректное `/api/shutdown`.

Если smoke-test релизного `.exe` нужно пропустить, например в ограниченном CI-окружении:

```powershell
.\build.ps1 -SkipSmokeTest
```

Или вручную тем же источником конфигурации:

```powershell
python -m PyInstaller --clean .\STO_CRM.spec
```

python -m py_compile .\sto_crm.py .\sto_crm\*.py .\sto_crm\assets\__init__.py .\tests\test_sto_crm.py

```

```

## Публикация обновления

Для публикации обновления в выбранный публичный репозиторий GitHub Releases загрузите только release-assets:

- `STO_CRM.exe` — готовая Windows-сборка;
- `STO_CRM.exe.sha256` — checksum;
- `latest.json` — manifest обновления.

Краткие заметки релиза публикуйте в описании GitHub Release, а не отдельным asset-файлом.

Минимальный `latest.json`:

```json
{
  "version": "1.17.1",
  "tag": "v1.17.1",
  "name": "СТО CRM 1.17.1",
  "asset": {
    "name": "STO_CRM.exe",
    "size": 12345678,
    "sha256": "...",
    "download_url": "https://github.com/markbakaa88/sto-crm/releases/download/v1.17.1/STO_CRM.exe"
  }
}
```

Именно `latest.json` читает раздел `Обновления` в CRM; приложение скачивает только опубликованные release-assets.
